*What is REST?
REST is not a protocol, but a stylr of developing web services that uses http methods like post, put, get calls to define actions.

*What is RESTful webservice?
these are no protocols like SOAP but  it uses HTTP methods/protocol to implement the concept  of REST architecture. Restful webservices define, URI(unified resource
identifier), provides resource representation like JSON and set of HTTP methods.

*Difference between URI(Uniform resource Identifier), URN(Uniform Resource Name) and URL(Uniform resource Locator):
from this path: http://thinkzarahatke.com/author/amty.html#post

URI: http://thinkzarahatke.com/author/amty.html#post
Format of URI is: <protocol>://<service-name>/<Resource-Type>/<ResourceID>
URN: //thinkzarahatke.com/author/amty.html#post
URL: http://thinkzarahatke.com/author/amty.html

*Most commonly used HTTP calls:

PUT: can only update the user information/resource.

PATCH: can update only specific information that we need to update. just update name. It'll not update entire object but a single entity.

DELETE: remove the specified resource.

HEAD: HEAD and GET both are same. but HEAD will just give the header information, it won't give u any response body(its a huge payload under body)

*** HTTP status code
2xx codes
200- success
201- any data has been created using POST call
202- Accepted
204- No content i.e when a data is deleted, the content is not found and hence the code is 204. Found during DELETE call.

3xx codes
304- Not Modified: if a doc is not modified

4xx codes
400- Bad Request: when a wrong JSON payload is sent, incorrect URI, comma missing
401- unauthorized: not authorised, authentication issue, token is expired
403- forbidden: server understood the request but cannot process/fulfil it as u r not authorized.
404- not found: sending req to serv and user is not available in DB then it'll send the error from server side that not found.

5xx codes
500- internal server error: server is crashed, or server is not up and down completely. Or unable to understand the request.

** What are different Authentications in API?
HTTP- Basic/Bearer, Digest, OAuth
Basic:
Basic authentication is an HTTP-based authentication approach and is the simplest way to secure REST APIs. It uses a Base64 format to encode usernames and passwords, both of which are stored in the HTTP header.
Here𠏋 an example of a Basic Auth in a request header:
Authorization: Basic bG9sOnNlY3VyZQ==

Bearer Auth:
Bearer authentication (also called token authentication) is an HTTP authentication scheme that involves security tokens called bearer tokens.
The client must send this token in the Authorization header when making requests to protected resources:
Authorization: Bearer <token>

OAuth:

The most common implementations of OAuth use one or both of these tokens instead:

access token: sent like an API key, it allows the application to access a user𠏋 data; optionally, access tokens can expire.
refresh token: optionally part of an OAuth flow, refresh tokens retrieve a new access token if they have expired. OAuth2 combines Authentication and Authorization to allow more sophisticated scope and validity control.


** Have you used Asertions in API? What are they?
Assertions allow you to validate HTTP responses in your API tests. For example, a basic assertion would ensure that a request to assertible.com returns a Status code 200 every time -- or that the homepage contains the text Assertible.

** What is API caching?
Caching is the ability to store copies of frequently accessed data in several places along the request-response path. The REST APIs are cacheable. It is one of its architectural constraints. When a client requests some resources, the request first goes through a cache and then to the server.
